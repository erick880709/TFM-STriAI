1. Arquitectura General

1.1 Diagrama de Capas

┌─────────────────────────────────────────────────┐ │ app.py (Entry Point) │ │ Router · Sidebar · Session State · Auth │ ├──────────────┬──────────────────────────────────┤ │ app/ui/ │ app/services/ │ │ (Streamlit) │ (Business Logic + ML Inference) │ │ │ │ │ P01-P14 │ auth_service triage_service │ │ login │ patient_service │ │ patient │ inference_service │ │ vital_signs │ audit_service │ │ clinical │ report_service │ │ ia_classif │ │ │ dashboard ├──────────────────────────────────┤ │ models │ app/data/ │ │ audit │ database.py (SQLite) │ │ users │ 12 tablas · Índices · Migración │ │ control_cb │ SEED SQL · row_to_dict() │ │ historico │ │ ├──────────────┴──────────────────────────────────┤ │ app/config/settings.py │ │ .env · YAML · Variables de entorno │ └─────────────────────────────────────────────────┘ Pipeline ML (externo): src/ → Código reutilizable del pipeline notebooks/ → Exploración y experimentos models/ → Artefactos serializados (.pkl) run_pipeline.py → Pipeline 14 pasos ejecutable

1.2 Patrón Arquitectónico

Layered Monolith con separación modular:

  • UI Layer (app/ui/): Componentes Streamlit sin estado. Cada pantalla es una función render_*() independiente.
  • Service Layer (app/services/): Lógica de negocio pura. Clases con inyección de db_path. Sin dependencia de Streamlit.
  • Data Layer (app/data/): Acceso a SQLite. Schema DDL, helpers row_to_dict(), migraciones incrementales, datos semilla.
  • Config Layer (app/config/): Carga de .env y variables de entorno.
Principio clave: Las páginas UI nunca acceden directamente a la BD; siempre pasan por los servicios.

3. Base de Datos

3.1 Motor y Configuración

# app/data/database.py
conn = sqlite3.connect(db_path)
conn.execute("PRAGMA journal_mode=WAL")    # Write-Ahead Logging
conn.execute("PRAGMA foreign_keys=ON")     # Integridad referencial
conn.row_factory = sqlite3.Row              # Acceso por nombre de columna
Decisiones: WAL mode permite lecturas concurrentes. Foreign keys ON evita huérfanos. Sin ORM — control total del SQL.

3.2 Conversión PascalCase ↔ snake_case

El schema SQL usa PascalCase (IdPaciente, NivelSugeridoIA). El código Python usa snake_case. La conversión es automática:

def _pascal_to_snake(name):
    s1 = re.sub(r'([a-z0-9])([A-Z])', r'\1_\2', name)
    s2 = re.sub(r'([A-Z]+)([A-Z][a-z])', r'\1_\2', s1)
    return s2.lower()

def row_to_dict(row):
    return {_pascal_to_snake(k): row[k] for k in row.keys()}
⚠️ Al añadir columnas al schema, usar PascalCase en SQL. La conversión a snake_case es automática al leer.

6. Flujo de Datos

6.1 Ciclo de Vida de un Triaje

1. POST paciente    → INSERT Paciente
2. POST triaje      → INSERT EventoTriaje (Estado=Registrado)
3. PUT signos       → INSERT/UPDATE SignosVitales → Estado: EnEvaluacion
4. PUT evaluación   → INSERT/UPDATE EvaluacionClinica → Estado: PendienteIA
5. POST inferencia  → INSERT PrediccionIA + ExplicacionSHAP → Estado: Clasificado
6. PUT validación   → UPDATE EventoTriaje → Estado: Validado
7. PUT cierre       → UPDATE EventoTriaje (FechaHoraCierre) → Estado: Cerrado

6.2 Flujo de Inferencia IA

Datos clínicos (SignosVitales + EvaluacionClinica)
    │
    ▼
_build_feature_vector()
    ├── Variables estructuradas → Scaler + Encoder
    ├── Embeddings NLP (BERT-es 768-dim)
    └── Vector combinado (Early Fusion)
    │
    ▼
model.predict_proba() → 5 probabilidades [I, II, III, IV, V]
    │
    ▼
_apply_thresholds() → Nivel sugerido (argmax calibrado)
    │
    ▼
SHAP TreeExplainer → Valores SHAP → Waterfall plot
    │
    ▼
_enrich_explanation() → Etiquetas clínicas en español

10. Guía para Desarrolladores

10.1 Añadir Nueva Página

  1. Crear app/ui/nueva_pagina.py con función render_nueva_pagina().
  2. Agregar ruta en app.py: elif page == "nueva_pagina": ...
  3. Agregar botón en el sidebar.
  4. Si requiere permisos: agregar a ROLE_PERMISSIONS en auth_service.py.
  5. Agregar prefijo a la lista de recarga de módulos en app.py.

10.2 Añadir Columna a la BD

  1. Agregar al CREATE TABLE en SCHEMA_SQL.
  2. Agregar ALTER TABLE ADD COLUMN a _apply_migrations().
  3. Usar PascalCase en SQL. La conversión snake_case es automática.
  4. Actualizar métodos del servicio (INSERT, SELECT).

10.3 Añadir Nuevo Rol

  1. Agregar a CHECK(Rol IN (...)) en CREATE TABLE Usuario.
  2. Agregar entrada en ROLE_PERMISSIONS.
  3. Actualizar ROLES_LABELS y ROLES_DISPONIBLES en UI.

10.4 Añadir Estado al Triaje

  1. Agregar a CHECK(Estado IN (...)) en CREATE TABLE EventoTriaje.
  2. Definir transiciones en TRANSICIONES_VALIDAS.
  3. Actualizar TriageService.transition_state() y UI.

10.5 Registrar Nuevo Modelo ML

  1. Entrenar con run_pipeline.py o manualmente.
  2. El artefacto se guarda en models/nombre_modelo/ con ModelSerializer.save().
  3. En la app: ⚙️ Gestión Modelos → Disco → Registrar en BD.
  4. Revisar métricas en 🔬 Comparar Modelos.
  5. Activar desde BD → Activar.

10.6 Debugging

  • Logs: Terminal donde se ejecuta streamlit run.
  • Errores de importación: Verificar lista de recarga en app.py.
  • BD: Se recrea al borrar data/triaje.db. Datos semilla con INSERT OR IGNORE.
  • Caché stale: Matar Streamlit, borrar __pycache__/, reiniciar con PYTHONDONTWRITEBYTECODE=1.
