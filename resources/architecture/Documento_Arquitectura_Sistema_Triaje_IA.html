Introducción y Objetivos ### 1.1 Propósito del documento Este documento describe la arquitectura de software del **Sistema de Triaje Multimodal basado en IA para servicios de urgencias en Colombia**. Está dirigido al equipo de desarrollo, a la directora del TFM y al tribunal evaluador de UNIR. Establece las decisiones arquitectónicas fundamentales, los diagramas de estructura (C4), los flujos de ejecución críticos y las consideraciones específicas del componente de Machine Learning. ### 1.2 Objetivo funcional Dado un conjunto de datos clínicos de un paciente que llega a urgencias (signos vitales, demográficos, antecedentes, texto libre de motivo de consulta), el sistema predice el nivel de triaje (I-V según Resolución 5596 de 2015 del Ministerio de Salud de Colombia), con salida explicable (SHAP), como **apoyo** — nunca reemplazo — al profesional sanitario que valida la clasificación. ### 1.3 Atributos de calidad (requerimientos no funcionales) | Atributo | Meta | Fuente | |---|---|---| | **Rendimiento** — Latencia de inferencia | < 3 segundos | RNP-001 | | **Disponibilidad** — Modo degradado | Clasificación manual si IA no disponible | RNO-006 | | **Seguridad** — Protección de datos | Ley 1581/2012, anonimización, TLS, RBAC | RNS-001 a 010 | | **Usabilidad** — Flujo de triaje | < 5 minutos para completar | OC-007, RNF-004 | | **Explicabilidad** — Cobertura XAI | 100% de predicciones con SHAP | RNX-001 | | **Trazabilidad** — Auditoría | Registro inmutable de cada acción | RNAU-001 a 006 | | **Mantenibilidad** — Arquitectura modular | Componentes desacoplados, configuración externalizada | RNF-007 | | **Desempeño del modelo** — F1-Score | ≥ 0.82 (macro-promedio) | RNF-006 | | **Desempeño del modelo** — AUC-ROC | ≥ 0.87 (macro-promedio) | RNF-006 | | **Desempeño del modelo** — Recall Niveles I-II | Priorizado sobre Precisión | RT-004 | --- ## 2. Estrategia de Solución ### 4.1 ¿Por qué Machine Learning? El triaje es un problema de clasificación multiclase (5 niveles ordinales) donde la relación entre variables predictoras (signos vitales, síntomas, demográficos) y el nivel de gravedad es no lineal, con interacciones complejas y patrones que no se prestan a un conjunto exhaustivo de reglas deterministas. Los sistemas de triaje estructurados existentes (MTS, CTAS, ESI) muestran variabilidad entre evaluadores humanos (Kappa ~0.43 reportado en Lidal et al. 2017). Un modelo de ML entrenado sobre grandes volúmenes de datos clínicos puede capturar estos patrones y proporcionar una recomendación objetiva, reproducible y explicable. ### 4.2 ¿Por qué multimodal? Los datos estructurados (signos vitales) y el texto libre (motivo de consulta) capturan información complementaria: los signos vitales cuantifican el estado fisiológico, mientras que el texto libre aporta el contexto narrativo que el paciente o el profesional expresan. La literatura muestra que la combinación de ambas modalidades supera consistentemente a los modelos unimodales (Levin et al. 2021: F1 0.81, +12% vs. unimodal). ### 4.3 División de responsabilidades: reglas vs. modelo | Responsabilidad | A cargo de | Ejemplo | |---|---|---| | **Predicción de nivel** | Modelo ML (XGBoost Early/Late Fusion) | SpO₂=88% + dolor torácico → sugiere Nivel II | | **Validación de rangos fisiológicos** | Reglas deterministas (frontend + backend) | Temperatura > 45°C → alerta antes de inferencia | | **Umbral de decisión clínica** | Reglas de post-procesamiento | Recall prioritario sobre Precision en Niveles I-II | | **Decisión final** | Profesional sanitario | Siempre prevalece sobre la sugerencia de IA | | **Seguridad y auditoría** | Reglas de negocio (RBAC, append-only) | No aplica IA | ### 4.4 Estilo arquitectónico **Monolito modular** con separación clara de capas (UI / Servicios / Datos / IA). Justificación: - El alcance es una demo/prototipo para TFM, no un sistema productivo con requerimientos de escalado independiente. - Streamlit unifica frontend y backend en un solo proceso, lo que simplifica el desarrollo y despliegue. - La separación en módulos Python independientes (`app/ui/`, `app/services/`, `app/data/`, `app/ia/`) garantiza que la migración a microservicios sea posible en el futuro sin reescribir la lógica de negocio. - El pipeline de entrenamiento es offline y completamente independiente de la aplicación demo. --- ## 5. ADR — Decisiones de Arquitectura ### ADR-001: Arquitectura monolítica modular para la demo **Contexto:** El sistema es un prototipo para TFM, no un producto productivo. Se necesita minimizar el esfuerzo de desarrollo y despliegue. **Decisión:** Monolito modular con Streamlit (UI + backend en un solo proceso Python), separación lógica por capas (UI / Services / Data / IA). **Alternativas consideradas:** - Microservicios: sobre-diseñado para el alcance. Agregaría complejidad de orquestación sin beneficio real. - Flask + frontend separado: más control pero duplica el esfuerzo de desarrollo de UI. **Consecuencias:** - Desarrollo más rápido (un solo archivo/deploy). - Migración futura a arquitectura distribuida posible gracias a la separación en módulos. - Limitación: Streamlit no soporta múltiples usuarios concurrentes de forma nativa — aceptable para demo. ### ADR-002: Streamlit como framework UI **Contexto:** El documento de contexto menciona Streamlit y Flask como alternativas, sin decisión firme. Ver `05-PENDIENTES-PARA-DIRECTORA.md`. **Decisión:** Streamlit (pendiente de confirmación por la directora). **Justificación:** - Integración nativa con visualizaciones Python (SHAP, matplotlib, plotly). - Menor esfuerzo para formularios + gráficos. - Un solo comando de arranque (`streamlit run app.py`). **Si se elige Flask:** Replanificar TT-E1-01 (inicializar proyecto) y las 12 pantallas (HU-E2-* y HU-E4-*). ### ADR-003: Priorizar Recall sobre Precisión en Niveles I-II **Contexto:** `RF-IA-003` originalmente describía argmax puro (seleccionar la clase con mayor probabilidad). La decisión técnica cerrada en `CONTEXTO_TRIAJE.txt` v2.0 y `02-ESPECIFICACION-TECNICA-MODELOS-IA.md` establece umbral optimizado. **Decisión:** Aplicar threshold tuning para maximizar Recall en Niveles I-II, usando argmax estándar para Niveles III-V. **Justificación clínica:** El costo de un falso negativo en Nivel I-II (subclasificar a un paciente crítico) es un riesgo vital inasumible. Un falso positivo (sobreclasificar) genera sobrecarga operativa pero no pone en peligro la vida del paciente. **Consecuencias:** Ligera caída de Precisión en Niveles I-II. Se documenta el punto de equilibrio elegido en el capítulo de resultados del TFM. ### ADR-004: SQLite como base de datos para la demo **Contexto:** La demo no requiere un motor de base de datos separado ni concurrencia multiusuario real. **Decisión:** SQLite (archivo local, sin servidor). **Alternativas:** - PostgreSQL: motor robusto pero requiere instalación y configuración separada — rompe el principio de "un solo comando de arranque". - MySQL: igual que PostgreSQL. **Consecuencias:** Migrar a PostgreSQL en producción requeriría cambiar el driver de conexión pero no el esquema. ### ADR-005: Late Fusion con Stacking **Contexto:** El método de combinación en late fusion está abierto (`05-PENDIENTES-PARA-DIRECTORA.md` #4). **Decisión tentativa:** Evaluar stacking con Regresión Logística como meta-clasificador (primera opción), y promedio ponderado como alternativa simple. **Justificación:** Stacking aprende pesos óptimos para cada submodelo a partir de los datos, en lugar de asumir pesos fijos. Es el enfoque más riguroso para un TFM. --- ## 16.
